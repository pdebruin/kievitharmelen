# PoC Plan: Fediverse Photo Poller + Moderation Queue + Gallery

## Problem
De Kievit Harmelen wants community-driven photo sharing without relying on Meta platforms. The Fediverse (Mastodon/Pixelfed) is a natural fit: people post photos with a hashtag, the website picks them up.

## Approach
Build an **API-first** Python (FastAPI) + SQLite backend. The PoC is a **standalone website** with potential WordPress integration later.

Two submission channels:
1. **On-website upload form** — visitors submit photos directly
2. **Fediverse hashtag poller** — automatically pulls in photos tagged `#DeKievitHarmelen`

Both feed into a **unified moderation queue**. Approved photos are published to:
- The **website gallery** (primary)
- **RSS/Atom feed**
- **Fediverse auto-post** (De Kievit's own Mastodon/Pixelfed account)
- **Share buttons**: Signal, email, copy link

### WordPress integration (future, not PoC)
The current dekievitharmelen.nl runs on WordPress. By building API-first, the site owners can later choose to embed the photo feed in WP via a JS widget or shortcode. This does not affect the PoC architecture.

## Stack
- **Python 3.11+** with FastAPI
- **SQLite** for storage
- **Jinja2** templates for the built-in web pages (keep it simple, no SPA)
- **HTTPX** for async HTTP calls to Mastodon API
- **APScheduler** or simple background task for polling
- **CORS middleware** — allow cross-origin requests so WordPress (or any site) can call the API

## Data Model

```sql
CREATE TABLE submissions (
    id TEXT PRIMARY KEY,
    source TEXT NOT NULL,          -- 'fediverse', 'upload', etc.
    source_url TEXT,               -- original post URL on Mastodon/Pixelfed
    source_instance TEXT,          -- e.g. 'mastodon.social'
    author_name TEXT,
    author_handle TEXT,            -- e.g. '@jan@mastodon.nl'
    author_avatar_url TEXT,
    description TEXT,
    created_at TIMESTAMP NOT NULL, -- when originally posted
    fetched_at TIMESTAMP NOT NULL, -- when we picked it up
    status TEXT DEFAULT 'pending', -- 'pending', 'approved', 'rejected'
    moderated_at TIMESTAMP,
    moderated_by TEXT
);

CREATE TABLE photos (
    id TEXT PRIMARY KEY,
    submission_id TEXT NOT NULL REFERENCES submissions(id),
    original_url TEXT NOT NULL,     -- URL on the source platform
    local_path TEXT,                -- local/stored copy
    alt_text TEXT,
    width INTEGER,
    height INTEGER,
    sort_order INTEGER DEFAULT 0
);

CREATE TABLE poller_state (
    instance TEXT PRIMARY KEY,
    last_seen_id TEXT,             -- track pagination
    last_polled_at TIMESTAMP
);
```

## Todos

### 1. project-setup
Set up Python project with FastAPI, dependencies, project structure.

### 2. data-model
Create SQLite database schema and basic CRUD operations.

### 3. json-api
Build the public JSON API (consumed by both our own gallery pages and external sites):
- `GET /api/photos` — paginated list of approved photos (filterable by date, album)
- `GET /api/photos/{id}` — single photo detail
- `GET /api/submissions/{id}` — submission with its photos (admin only)
- CORS headers so any external site can call the API later
- OpenAPI docs auto-generated by FastAPI at `/docs`

### 4. upload-form
Build the on-website photo submission form ("Deel je foto's"):
- Multi-file upload (max ~10 photos, jpg/png/webp, max 15MB each)
- Name + email fields (required)
- Description (optional)
- GDPR consent checkbox
- Auto-resize/compress images on upload
- Spam prevention: honeypot field, rate limiting
- Submitted photos go into moderation queue as `source='upload'`

### 5. fedi-poller
Build the Fediverse hashtag poller:
- Poll `GET /api/v1/timelines/tag/dekievitharmelen` on configured instances
- Parse response, extract media attachments
- Download and store images locally
- Insert into submissions + photos tables
- Track last_seen_id to avoid duplicates
- Run on a schedule (every 10-15 min)

### 5a. mock-fedi-server
Build a mock Mastodon API server for testing (see `test-plan.md`):
- `POST /api/v1/media` + `POST /api/v1/statuses` — simulate publishing posts with photos
- `GET /api/v1/timelines/tag/{hashtag}` — serve posts in Mastodon API format
- In-memory storage, no persistence — fresh state per test run
- Includes a test client (`tests/fedi_client.py`) for publishing test posts with sample images

### 6. admin-queue
Build the moderation queue page:
- List pending submissions with photo thumbnails
- Approve/reject buttons (per photo or per submission)
- Simple auth (hardcoded token or basic auth for PoC)
- Show counts: pending / approved / rejected

### 7. public-gallery
Build the public gallery page:
- Built-in standalone HTML pages that consume our own JSON API
- Display approved photos in a grid
- Show attribution (author name/handle, link to original post)
- Lightbox for full-size viewing
- Sorted by date (newest first)
- Share buttons on each photo: Signal, email, copy link

### 8. distribution
Set up outbound distribution for approved photos:
- **RSS/Atom feed** at `/feed` — new approved photos
- **Fediverse auto-post** — post approved photos to De Kievit's own Mastodon/Pixelfed account via API

### 9. integration-test
Test the full pipeline end to end:
- Post a photo with the hashtag on a Mastodon/Pixelfed instance
- Verify it appears in the moderation queue
- Approve it, verify it shows on the public gallery

See `test-plan.md` for full test scenarios (T1-T11).

## Key Decisions
- **API-first architecture** — JSON API is the primary interface; standalone HTML pages consume the same API
- **Standalone PoC** — no WordPress dependency; WP integration is a future option via JS widget/shortcode
- **Two submission channels** in PoC — website upload form + Fediverse hashtag poller
- **Three distribution channels** — website gallery, RSS feed, Fediverse auto-post
- **Share buttons** — Signal, email, copy link (no Meta platforms without explicit approval)
- Fediverse API is public, no auth needed for reading public hashtag timelines
- Download images locally rather than hotlinking (respect source servers, survive deletions)
- Simple server-rendered HTML (Jinja2) for standalone pages, no JavaScript framework needed
- CORS enabled so the API can be consumed by external sites later
- Poll multiple instances (mastodon.social, mastodon.nl, pixelfed.social) to cast a wide net
